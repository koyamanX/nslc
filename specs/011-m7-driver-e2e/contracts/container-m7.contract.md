# Contract: `:dev-m7` dev container surface

**Feature**: 011-m7-driver-e2e
**Owners**: `docker/Dockerfile.dev`, `docker/publish-images.yml`,
`scripts/ci.sh`
**Status**: Frozen at M7
**Related FRs**: FR-026
**Related contracts**: `audited-corpus.contract.md` (consumer)

---

## §1 Image identity

- **Tag**: `ghcr.io/koyamanX/nsl-nslc:dev-m7`
- **Parent**: `ghcr.io/koyamanX/nsl-nslc:dev` (the current rolling
  tag at the time M7 lands)
- **Build mechanism**: `docker buildx` via
  `docker/publish-images.yml` workflow, using the established
  `PARENT_IMAGE` build-arg pattern documented in the
  `project_publish_images_buildx_isolation.md` memory entry. The
  build-arg is passed via the workflow input
  `--build-arg PARENT_IMAGE=ghcr.io/koyamanX/nsl-nslc:dev` —
  buildx-container isolation requires this; `docker tag`-based
  aliasing does NOT work.

The `:dev-m7` tag is non-rolling for the duration of the M7 PR's
review cycle. Once M7 merges, a follow-on PR bumps the `:dev`
rolling tag to point at the same image SHA `:dev-m7` resolved to.

## §2 Additions over parent (`:dev`)

The M7 `Dockerfile.dev` adds:

| Tool | Pin | Source |
|---|---|---|
| Verilator | `v5.024` | `git clone --branch v5.024 --depth 1 https://github.com/verilator/verilator.git`, autoconf+configure+make |
| `riscv32-unknown-elf-gcc` | `13.2.0` | `apt install gcc-riscv64-unknown-elf` on the base Debian/Ubuntu (then symlink-rename for rv32) OR build-from-source if apt lags |
| `riscv-tests` binaries | upstream master SHA pinned in `docker/publish-images.lockfile.yml` | `git clone https://github.com/riscv-software-src/riscv-tests.git`, build the `isa/` suite |

Pins are recorded in `docker/publish-images.lockfile.yml` (the
parallel publish-images workflow's per-tag lockfile). After the
first successful publish, the lockfile records the resolved
commit SHAs so subsequent publishes are bit-reproducible.

**Removed/replaced from parent**: nothing. M7 is purely additive
over `:dev`.

## §3 Path additions

Inside the container:

- `verilator` available on `$PATH` (typically `/usr/local/bin/verilator`).
- `riscv32-unknown-elf-{gcc,objcopy,objdump,as,ld}` on `$PATH`.
- `riscv-tests` build outputs under `/opt/riscv-tests/` with
  symlinks for the per-instruction `.elf` binaries.

No new environment variables exported by default beyond what
`:dev` already exports.

---

## §4 CI pin policy

`scripts/ci.sh` is parameterized by a container-tag variable
(currently `:dev`). At M7 PR's CI:

- The lit-test cell that runs `check-audited` MUST be pinned to
  `:dev-m7`.
- Other CI cells (clang-format / clang-tidy / SPDX / static-checks
  / build) MAY continue to use `:dev` (they have no Verilator
  dependency). Pinning ALL cells to `:dev-m7` is acceptable but
  not required.
- The container tag for each cell is recorded in `scripts/ci.sh`
  via a per-cell variable so the assignment is reviewable.

Post-merge follow-on PR: bumps `:dev` to the same image SHA
`:dev-m7` resolves to, then migrates all CI cells back to `:dev`.
After that PR lands, `:dev-m7` is retained for ~30 days as a
bisection aid, then deleted.

## §5 Test gate for the image bump itself

Before M7 PR can pass CI, the publish-images workflow MUST have
succeeded for `:dev-m7` — verified by the workflow's own output
attestation. The M7 PR's CI does NOT trigger an image rebuild;
it consumes the already-published `:dev-m7` tag.

A separate small smoke test (`scripts/test_container_m7.sh`)
verifies post-image-bump that all three tools (`verilator`,
`riscv32-unknown-elf-gcc`, `riscv-tests/isa/rv32ui-p-add`) are
present and runnable inside the container. This smoke test runs
as the first step of the `check-audited` lit cell — if it fails,
the cell fails fast with a clear "container missing M7 deps"
message rather than a confusing per-fixture compile error.

## §6 Lockfile schema (`docker/publish-images.lockfile.yml`)

```yaml
# Auto-generated by docker/publish-images.yml at publish time.
# DO NOT hand-edit — re-publish the image to refresh.

images:
  - tag: ghcr.io/koyamanX/nsl-nslc:dev-m7
    digest: sha256:<64-char-hex>
    parent_tag: ghcr.io/koyamanX/nsl-nslc:dev
    parent_digest: sha256:<64-char-hex>
    published_at: <ISO-8601>
    tools:
      verilator:
        tag: v5.024
        commit: <git-sha>
      riscv_gcc:
        version: 13.2.0
        source: apt
      riscv_tests:
        commit: <git-sha>
```

The lockfile lands in the M7 PR alongside the `Dockerfile.dev`
edits. Subsequent re-publishes update the lockfile (digest +
published_at + per-tool commits) but the schema is stable.

## §7 Forward compatibility

This contract is frozen at M7. Anticipated future image-surface
changes:

- M8 may add riscv-formal Python deps + symbiyosys for the formal
  cell. That lands as `:dev-m8` via the same PARENT_IMAGE pattern
  off `:dev` (now `:dev` will resolve to what was `:dev-m7` after
  the follow-on rolling-tag bump).
- M9 may add release-pipeline tooling (e.g., `goreleaser`-equivalent
  for C++ binaries; signed-artifact tools). Out of M7 scope.

The dev-container surface is project-side ergonomic infrastructure;
it does not appear in user-facing artifacts. Changes to it are
governed by routine PR review against this contract, NOT by
/speckit-clarify.
